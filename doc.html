🎲 Dicey Desk Duels — Developer Reference

The deep companion to README.md and CLAUDE.md. Written for anyone (human or AI) changing this codebase. Line numbers are anchors as of July 2026 — if one has drifted, grep the named symbol instead.

Overview & quickstart

A dice roguelike PWA: roll dice, poker-style combos multiply the sum into damage, fight across a procedurally generated 20-column branching map, bank beans between runs in a permanent Workshop economy. Vanilla JS, zero dependencies, no build step — the files you see are the files that ship. Live at dice.ahcomputing.com.

The one combat quirk everything is built around: a rolled 1 strikes as a 6 for the damage sum, but stays a 1 for combo detection (evaluate(), engine.js:39). A die of all 1s is both a heavy hitter and an automatic five-of-a-kind.

# serve (a PWA needs http(s); file:// won't register the service worker)
python3 -m http.server 8080          # → http://localhost:8080

# verify wiring after ANY content/art/view-id change — catches silent breakage
node validate.js

# balance harness: auto-plays N runs per tier, prints win% / avg node / avg beans
node sim.js 200

# headless engine smoke test (engine.js is DOM-free and self-requires content.js)
node -e "const E=require('./engine.js'); console.log(E.evaluate([6,6,6,1,1]))"

Architecture

Three-layer split, enforced by convention and worth defending fiercely:

FileLinesRoleHard rule
content.js~684All game data: one CONTENT objectPure data. No logic, no DOM. Adding a monster/upgrade should touch only this file.
engine.js~1034All game logic, exported as EngineNo DOM, no timers, no rendering. Must stay require()-able in node.
view.js~1215DOM, input, animation, skinsThe ONLY file that touches document. No game rules.
audio.js~207Synthesized SFX (Sound)Never touches game logic; view calls it. Node-safe no-op.
music.js~261Generative layered music (Music)Same; shares audio.js's one AudioContext via Sound.context().
sim.js~145Headless balance bot (node only)Not loaded by the page. Can drift from the engine API — sanity-check before trusting.
validate.js—Content/art/wiring linter (node only)Run after every content/art/id change. ERROR = broken now; WARN = drift/staging.

Load order is load-bearing

index.html loads plain (non-module) scripts in this exact order: content.js → engine.js → audio.js → music.js → view.js. Each IIFE reads earlier globals at eval time: engine reads window.CONTENT; view reads window.CONTENT and window.Engine (hard requirement — reorder and everything is undefined) and window.Sound/window.Music (optional — stubbed with no-ops at view.js:9–10, so audio/music can be deleted without breaking the game).

Every file dual-exports: window.X in the browser, module.exports in node. Keep that pattern — it is what makes validate.js/sim.js/headless tests possible.

Communication model

No event bus, no framework. The view calls an Engine function, the engine mutates state and often returns a list of events/hits, and the view reads the mutated state and animates the returned events (e.g. attack() returns {ev, hits, allDead, heal, coin, spiked, dodged}; enemyTurn() returns {actions, playerDead, allDead}).

Phase FSM

state.phase values: map (choose next node) · player (your turn in a fight) · reward · rest · treasure · shop · reforge · event · miniboss (leave-or-continue decision) · win · lose · left. The engine sets it; the view's enterPhase()/render switch on it. There is no enemy phase — the view calls enemyTurn() then advanceAfterEnemy() explicitly between player turns.

Migration model is schema-merge, not versioned. Loading does Object.assign(E.newBank(), saved) then backfillBank() — new fields need a default in newBank() (engine-owned) or backfillBank() (view/cosmetic). Nothing reads the .v1 suffix; a breaking schema change means bumping the key and writing a real migration by hand. Run state (player/run/enemies) is never persisted — a reload abandons the run.

Enemy schema

key: { icon:'📎', name:'Paperclip', hp:6, atk:3, gold:3, role:'grunt', …roleParams }

Role params (consumed in setIntent/enemyTurn): hits (swarm), heal (mender, boss bossSummonHeal), selfArmor (turtle), rage+startRage (berserker), wardArmor (warden), summon+summonEvery (summoner; board capped at 4 living), auraAtk (standard), hex (hexer), rust (rust), poisonCast (poisoner), poisonHit (venom-on-hit, any role), poisonAura (stack/turn while alive), startArmor (spawns armored), dodge (miss chance vs the primary hit), goldSteal (steals run beans on hit), buffer (boss-only: support add spawned alongside), bossSummonHeal (mold-colossus cycle), deathBurst (on-death chip damage to the player, floored at 1 HP — can't kill; deathEffects in engine), deathPoison (on-death poison splash — CAN kill later via the tick), armorDecay (armor melts N/round in advanceAfterEnemy).

Scaling: hp/atk are multiplied by (1 + BALANCE.xPerUpgrade × player.upgrades + greed) then by the challenge tier's enemy factor (engine.js:60, 271). gold is multiplied by player.goldMult.

Die schema

{ faces:[1,2,3,4,5,6], value, feature:null, flevel:0, _rr, _mom, anchor?, split? }

Faces are numbers 1–6 or 'W' (wild: any value for combos, exactly 1 for the sum). Face mods are pure face-array ops in Engine.EFFECTS: forge/load/brand/engrave/uniform/polish. feature+flevel is the one feature slot (same feature again → level up; different → replace, see applyFeature engine.js:643). Max 5 dice; max 4 living enemies on the board. Load raises the 2 lowest faces, Engrave the 3 lowest (shared raiseLowest helper); Brand (all faces → 1) is a treasure premium, excluded from the Reforge workbench, which itself allows at most 2 face-mod stamps per visit (state.reforge.mods).

view.js / audio.js / music.js essentials

  • Art layer (view.js:13–43): every sprite is artImg(category, key, fallbackEmoji, opts) → <img src="art/<category>/<key>.png"> with onload/onerror handlers on window.__UFart. Mode replace (enemies, hero): 404 → the emoji text node. Mode layer (die bodies): 404 → img removes itself, leaving the plain die + icon badge. Categories in use: enemy, skin, die, plus the hard-coded art/banner/kitchen_coming_soon.svg.
  • SKINS (view.js:45): 36 entries {id, emoji, name, unlock, mult?|cost?|hint?}; unlock ∈ free | default | challenge | shop | condition. Ownership lives in bank.skins; the equipped one in bank.skin.
  • Debuff label maps (view.js:1073): DEBUFF_FX (hero flash text) and INTENT_DEBUFF (enemy telegraph label). A new debuff kind must be added to BOTH or it renders as a bare ✸.
  • Sound: S.play(name) with names from audio.js's FX: roll, lock, attack, enemyHurt, playerHurt, heal, shield, win, lose, button, purchase — plus S.combo(baseMult), the combo-tier arpeggio keyed off res.ev.baseMult (the raw combo mult before kindle/prism bonuses).
  • Music: M.start/stop/setBiome('desk')/enterMiniboss/enterBoss/bossPhase(2)/clearBoss. Biome palettes live in music.js:22, structured so a kitchen palette drops in. Boss intensity = extra layers on the same 16th-note clock, not a new track.
  • Playtest stats (view.js:135): every run end calls recordRun(result); exportable JSON aggregate — useful for tuning.

Recipe: add an enemy

Also a skill: /add-enemy (.claude/skills/add-enemy/SKILL.md) — if this recipe changes, update the skill in the same commit.

  1. content.js ENEMIES: add mykey: { icon, name, hp, atk, gold, role } — role must exist in ROLES; add the role's params (e.g. hits for swarm). Look at a same-role neighbor for stat ranges at that depth.
  2. Drop its id into a band of ENCOUNTER_POOLS/_S2/_KITCHEN (or ELITE_POOLS*) — otherwise it never spawns.
  3. Optional: a PUNS.mykey battle-start line; falls back to PUN_FALLBACK.
  4. Optional: art/enemy/mykey.png (transparent square, ~256×256) + a line in art/enemy/MANIFEST.txt. No PNG = emoji fallback, by design.
  5. Bosses: role:'boss', listed in MAP.bossKeys (or MAP_KITCHEN.bossKeys) instead of pools; optional summon (feeds Reinforcements/Lifelink) and buffer (support add). Signatures are drawn at random — a boss-specific forced signature is an engine edit (see the mousetrap/windup pattern, engine.js:279–284).
  6. Run node validate.js — it catches bad roles, unknown pool/summon ids, and manifest drift.

Recipe: add a die feature

Also a skill: /add-feature (.claude/skills/add-feature/SKILL.md) — if this recipe changes, update the skill in the same commit.

All die-feature behavior lives in ONE registry: Engine.FEATURE_HOOKS (engine.js, right above rollDie — its header comment documents every hook signature). A die carries exactly one feature, so each hook is a single dispatch. validate.js enforces a 1:1 pairing between CONTENT.FEATURES and FEATURE_HOOKS — an unwired feature is a hard ERROR, not a silent no-op.

  1. content.js FEATURES: add mykey: {icon, name, desc} (icon doubles as the die badge fallback).
  2. content.js UPGRADES: add { id:'mykey', name:'… (die)', desc, effect:'addFeature', feature:'mykey' } so it can be offered.
  3. engine.js FEATURE_HOOKS: add one entry implementing the right hook(s):
    HookFiresExamples
    onRoll(d, v, rng, floor) → vwhenever the die rollsecho, ascend
    onTurnStart(state, p, d)fresh turnbulwark, momentum stack
    onReroll(state, p, d, rng)after a manual reroll of this diemagnet, bulwarkRoll + siphonRoll (≤5/turn), momentum reset
    damage(p, d, acc)damage evaluation (preview == attack automatically)whetstone (acc.sum), kindle (acc.kindle, capped +1), prism/overcharge/momentum (acc.mult), banker (acc.flat)
    onAttack(state, p, d, ctx, acc)attack resolution; ctx {ev, dealt}siphon* (acc.heal), bulwarkCombo, overflow, jackpot
    splash(d, mult) / aoe(state, p, d, ctx)attack AoEcleave / shockwave, bubble, ricochet
    reflect(p, d) / counterBank(d)when an enemy hits youthorns, thornsCombo / counter
    pierce: true / overkill: truedeclarative flags read by attack()piercer / overkill
  4. Levels: d.flevel scales the effect (picking the same feature again levels it). Honor it.
  5. Optional: add the id to fitting CLASSES[].ids; art/die/mykey.png + MANIFEST line.
  6. Verify: node validate.js (ERRORs on a missing registry entry), node sim.js 100, and check the in-fight preview equals the dealt damage.

Recipe: add a skin

  1. view.js SKINS: add { id, emoji, name, unlock: … }. Optional art/skin/id.png + MANIFEST line.
  2. Unlock wiring by type:
    • challenge (+mult: N) — nothing else; granted automatically on a winning run at that tier.
    • shop (+cost: N) — nothing else; the Workshop shop lists it automatically.
    • condition (+hint) — three-file change: a threshold in content.js SKIN_UNLOCKS, a grant('id', condition) line in Engine.evaluateSkinUnlocks (engine.js:956), and usually a persistent counter field in newBank() (engine.js:66) that something increments (see beetleKindnessCount). Purely view-detectable conditions can instead call view.js unlockSkin('id') directly (gremlin, headless pattern, view.js:118).
    • event-granted — an EVENTS choice with outcome:'unlockSkin', skin:'id' (robot, fox pattern).
  3. node validate.js verifies every condition skin has a grant path and every id has art/manifest coverage.

Recipe: add an enemy status effect (the poison pattern)

Poison is the template for any tick-based debuff. Copy its five touch points:

  1. State field: initialize on player in startRun (engine.js:234 — poison: 0 lives with the other accumulators).
  2. Application sites in enemyTurn: per-hit (ea.poisonHit, engine.js:595), cast intent ({type:'debuff', debuff:'poison', value} set in setIntent, applied engine.js:607), passive aura (engine.js:616), and/or event-queued next-fight debuff (applyPendingDebuffs, engine.js:792).
  3. Tick at the top of startPlayerTurn (engine.js:409): apply damage, then — critically — if it kills, call loseRun(state) and return. Routing death through state.phase='lose' centrally is what lets both sim and view resolve it; the view re-checks phase==='lose' after the enemy turn (view.js:~618) specifically to avoid the between-turns softlock.
  4. Cleanup in winFight (engine.js:982): zero the stack / restore modified stats so nothing leaks across fights.
  5. View labels: add the kind to BOTH DEBUFF_FX and INTENT_DEBUFF (view.js:1073–1074), or it renders blank; add a flash/status-panel readout if it persists (poison's is at view.js:~620 and ~1018).

Recipe: add an event · add a sound

Event: add an entry to content.js EVENTS using an existing outcome id — the full menu (riskBite, raid, timedBuff, trickleIncome, stateScaledReward, scoutReveal, randomTable, unlockSkin, plus the simple pay-X-for-Y ones) is documented in the comment block at content.js:497 and covers most designs without engine work. Gate deep-run setpieces with minCol; hide a choice's outcome with hidden:true. A genuinely new outcome = one branch in Engine.applyEvent (engine.js:854; set state.phase and return a message) — reuse resolveOutcomeBlock for standard payouts.

Sound: add a synth function to audio.js FX (compose tone()/noise(); study neighbors), then call S.play('name') from view.js at the event point. Music reacts via the M.* API — don't call Sound/Music from engine.js, ever.

Recipe: wire the kitchen biome (staged, not wired)

Current state: ~50 kitchen enemies, ENCOUNTER_POOLS_KITCHEN, and MAP_KITCHEN (minibosses fixed at cols 5/10/15 via bossKeys{5,10,15,boss}, boss at col 20) exist as data. Poison and the new role params are fully engine-implemented. Not yet built:

  • Map plumbing: generateMap(rng, content) already accepts a content override for pools, but reads map params from (content||C).MAP — it must learn to take MAP_KITCHEN's shape (multiple fixed miniboss columns with per-column boss keys, boss at col index 20 → 21 columns) and pick pools by biome. Suggested approach: a biome arg that selects {MAP, pools}, keeping the desk path byte-identical.
  • Run flow: how a run reaches the kitchen (post-desk continuation? a Workshop selector?) — a design decision, not code.
  • View: biome-aware backdrop/banner (the art/banner/kitchen_coming_soon.svg teaser is hard-coded at view.js:~195).
  • Music: add a kitchen palette to music.js PALETTES (music.js:22 — the structure anticipates it) and call M.setBiome('kitchen') on entry.
  • Art: all 50 kitchen enemies currently run on emoji fallbacks (validate.js WARNs). The bestiary advises shipping only ~12–15; kitchen_bestiary.md is the design source.
  • Mechanics are complete: poison, and (as of July 2026) deathBurst/deathPoison/armorDecay are all engine-implemented with view feedback — the biome needs only map wiring + art + music.

Recipe: ship a change (PWA cache discipline)

Also a skill: /ship (.claude/skills/ship/SKILL.md) — if this recipe changes, update the skill in the same commit.

  1. Run node validate.js and node sim.js 100 (after balance changes).
  2. Bump const CACHE = 'dice-duels-vN' in sw.js (sw.js:2). Online clients get fresh code anyway (shell is network-first), but the bump purges old caches and is what updates offline/installed clients.
  3. New non-art file that the app needs offline? Add it to the SHELL list (sw.js:3). Never add art/ PNGs to SHELL — one 404 rejects the whole addAll() and the PWA silently never installs; art is runtime-cached cache-first instead (sw.js:32).
  4. "My change doesn't show up" during dev = stale SW cache. Hard-reload, or DevTools → Application → Service Workers → Update/Unregister. Suspect this before suspecting the code.

More traps:

  • Preview must equal attack. The damage readout and the actual hit share effEval — if you add damage logic in attack() without featureBonus/effEval awareness, the preview lies.
  • Turning every reward off falls back to the full pool (enabledPool, engine.js:709) — "all disabled" ≠ "no rewards".
  • Combo multiplier is globally capped at 5× (effEval, engine.js:467); board capped at 4 living enemies; hand capped at 5 dice.
  • addBeans is the only legal way to grant beans — bypassing it skips lifetimeBeans and skin-unlock checks.
  • Engine functions assume state.bank exists (skin unlocks, bean accrual). sim.js supplies one; so must any new harness.
  • view.js and index.html cannot be require()d — validate.js parses them as text; if you restructure the SKINS array or audio's FX block, check validate.js's extraction regexes still match.
  • sim.js is not auto-synced with the engine — after renaming/removing an Engine function, grep sim.js.

Balance tuning

Also a skill: /balance (.claude/skills/balance/SKILL.md) — if this section changes, update the skill in the same commit.

  • Global difficulty curve: BALANCE.hpPerUpgrade/atkPerUpgrade — additive enemy scaling per reward the player takes (currently 0.12/0.11; history: 0.18/0.12 → 0.11/0.11 when the map grew to 20 nodes, → 0.12 HP after the July 2026 upgrade buffs). This is the single most sensitive knob in the game.
  • Workflow: change a knob → node sim.js 300 → compare win% across tiers (bare/mid/full/classes). The bot is a deliberate lower bound (cautious, skips reforge) — read results as a relative curve, not a skill ceiling. Human data: the playtest stats export in the Workshop (view.js:135).
  • Local knobs: per-enemy stats and pool composition (content.js), CHALLENGE tier factors, SHOP cost curves, RUN bonuses, NODECOST, MAP weights/gates.
  • Known placeholders to tune with real data: SKIN_UNLOCKS.cowboyLifetimeBeans (50 000) and SKIN_UNLOCKS.vikingTurnDamage (500), both marked PLACEHOLDER in content.js.
  • Design intent and locked decisions live in dice_roguelike_design.md (status legend: LOCKED / OPEN / TO BRAINSTORM); kitchen plans in kitchen_bestiary.md.
